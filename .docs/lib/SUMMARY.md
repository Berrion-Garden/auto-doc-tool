## Architecture Overview
`AutoDoc` is a modular, pipeline-driven software documentation generation tool. It follows a **layered architecture** that separates concerns into distinct phases: code analysis, data transformation, workflow orchestration, and documentation generation. The system leverages static analysis, structural graph building, and AI/LLM capabilities to produce comprehensive, multi-format documentation (diagrams, markdown, architecture guides, and READMEs) from a target codebase.

## Major Modules & Responsibilities
- **Analyzer:** Handles low-level codebase introspection. Extracts models, imports, YARD comments, schema definitions, and diffs. Identifies orphaned files and caches results to avoid redundant parsing.
- **Transformer:** Converts raw parsed data into structured, queryable domain models. Builds class hierarchies, container data flows, ERD relationships, file trees, and graph representations.
- **Orchestrator:** Drives the end-to-end generation pipeline using a step-based execution pattern. Coordinates sequencing, tracks metrics, and manages dependencies between analysis, transformation, generation, and AI enrichment phases.
- **Generator:** Produces final documentation artifacts. Includes specialized generators for C4 diagrams, class diagrams, ERDs, architecture markdown, summaries, index pages, and vector embeddings for search. Uses template helpers for consistent output formatting.
- **LLM Integration:** Provides a unified interface to Large Language Models. Handles prompt construction, client communication, response parsing, text summarization, and semantic enrichment to augment static analysis with AI-driven insights.
- **Reporter & Search:** Validates documentation completeness and generates audit reports. Maintains a centralized documentation index and powers query/search capabilities over generated assets.
- **Infrastructure & Utilities:** CLI interface, HTTP server for serving docs, YAML configuration loader, end-to-end test runner, and shared helpers for markdown formatting, output styling, timestamp tracking, and file tree operations.

## Module Relationships & Data Flow
The architecture follows a **bottom-up data pipeline**:
1. **Analysis → Transformation:** The `Analyzer` extracts raw structural and semantic data from the source code. This data is consumed by the `Transformer`, which normalizes it into interconnected domain models (graphs, hierarchies, flows).
2. **Orchestration:** The `Orchestrator` acts as the central conductor. It invokes the `Analyzer` and `Transformer`, triggers `LLM` enrichment where needed, and sequences execution steps to produce intermediate artifacts (manifests, vector indices, draft content).
3. **Generation → Delivery:** The `Orchestrator` passes structured data to specialized `Generator` modules, which render final documentation files. The `Reporter` audits output quality, while the `Server` and `SearchService` expose the results to users.
4. **Cross-Cutting Support:** `Config` and `Utils` provide shared infrastructure to all layers. The `LLM` subsystem can be called by either the `Orchestrator` or `Generator` to inject AI-generated summaries, architectural context, or enriched metadata.

## Key Architectural Patterns
- **Step-Based Pipeline:** The `Orchestrator` uses a base step abstraction to declaratively sequence documentation tasks, enabling easy extension and testing.
- **Separation of Concerns:** Parsing, transformation, orchestration, and output generation are strictly decoupled, allowing independent updates to analysis engines, AI providers, or output formats.
- **Caching & Incremental Updates:** The `AnalysisCache` and `DiffService` enable efficient re-runs by only processing changed or newly discovered code segments.
- **AI-Static Hybrid Analysis:** Combines deterministic code parsing with LLM-driven summarization and enrichment, balancing precision with contextual understanding.
- **Extensible Output Layer:** The generator suite follows a strategy pattern, allowing new documentation formats (e.g., additional diagram types or export targets) to be added without modifying core logic.
