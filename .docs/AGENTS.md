## Project Overview

**auto-doc-tool** is an automated documentation generator designed to analyze software projects and produce comprehensive technical documentation. By parsing source code and configuration files, the tool generates essential assets such as architecture diagrams (C4, class, and ER diagrams), README files, and searchable documentation indexes. It leverages both static analysis and Large Language Model (LLM) integration to provide AI-enhanced insights, ensuring that project documentation remains synchronized with the underlying codebase across multiple programming languages.

The project's architecture is organized into several distinct modules within the `AutoDoc` namespace. The **Analyzer** module (`lib/auto_doc/analyzer`) handles source code parsing, schema extraction, and dependency detection. The **Generator** module (`lib/auto_doc/generator`) is responsible for creating visual diagrams and Markdown documentation files. Additionally, the **LLM** module (`lib/auto_doc/llm`) manages AI interactions for content summarization, while the **Orchestrator** (`lib/auto_doc/orchestrator`) coordinates the overall documentation pipeline, ensuring that analysis and generation steps execute in the correct sequence.

## Core Architecture and Component Interaction
The `auto-doc-tool` implements a modular, multi-stage pipeline architecture designed to automatically generate comprehensive documentation from heterogeneous source codebases. The system is anchored by a central orchestrator that coordinates a sequence of specialized processing stages, each encapsulated within dedicated modules. At the foundation, an analyzer layer extracts structural and semantic metadata by parsing source files, reading existing documentation annotations, and interpreting database schemas and model associations. This extracted data flows into a transformer layer that constructs relational graphs, class hierarchies, and data flow models, effectively normalizing disparate code structures into a unified intermediate representation. The relationship between these layers is strictly unidirectional during processing: raw code is parsed, transformed into structured graphs, and then handed off to downstream consumers.

## Generative Workflow and Design Patterns
Once the semantic graph is established, the orchestrator dispatches the enriched data to a generator module that produces targeted documentation artifacts, including architecture overviews, C4 diagrams, ERDs, class diagrams, and vector-indexed summaries. To augment machine-generated content with natural language context, the architecture integrates a dedicated LLM component that abstracts API communication, prompt templating, and response parsing. The workflow prominently employs a step-based pipeline pattern, where each documentation phase is implemented as a reusable step inheriting from a base class, enabling conditional execution and predictable state progression. Performance optimization is handled through caching and diff-tracking services that prevent redundant analysis, while a dedicated reporter module applies completeness auditing and coverage metrics to validate output quality before delivery.

## Entry Points and System Integration
The architecture exposes dual execution interfaces via a command-line interface and a lightweight web server, supporting both interactive development workflows and programmatic CI/CD integration. Configuration management, output formatting, file tree construction, and timestamp tracking are centralized in utility modules that decouple infrastructure concerns from business logic. This strict separation of analysis, transformation, generation, and reporting phases follows a service-oriented design, allowing the core pipeline to remain stable while swapping out language-specific parsers, alternative vectorization strategies, or custom diagram renderers. The comprehensive test suite and fixture-driven evaluation structure further demonstrate a contract-first development approach, ensuring that each pipeline stage maintains predictable I/O boundaries across evolving documentation requirements.
