# Auto-Doc Tool — Product Requirements Document

## Product Vision

Auto-Doc transforms any codebase into a richly searchable, agent-friendly documentation library. Developers and AI agents alike can browse, search, and query code structure without ever grepping raw source files. The tool generates hierarchical index files, semantic search embeddings, structured summaries, and rich dependency diagrams — all from a single command. The primary consumer is the AI agent, which should never need to read raw source just to understand module structure, public API surfaces, or dependency relationships.

## User Roles

| Role | Description | Primary Goals |
|------|-------------|---------------|
| Developer | Writes and maintains code, uses auto-doc to generate/refresh docs | Generate docs quickly, verify coverage quality, commit reviewed drafts |
| AI Agent | Automated assistant that queries docs to understand codebase structure | Find module APIs, resolve dependencies, locate symbols without grep |
| CI Pipeline | Automated build system that gates PRs on documentation quality | Enforce coverage thresholds, detect orphan files, fail on drift |
| Maintainer | Gem author who ships releases and fixes bugs | Ensure all specs pass, E2E self-test validates the pipeline |

## Functional Requirements

### FR-1: CLI End-to-End Generation
**As a** developer,
**I want** to run `autodoc generate --directory <path>` and get a complete `.autodoc/` output tree,
**So that** I can produce documentation for any Ruby project with one command.

**Acceptance Criteria:**
- Running `generate` on a project with a valid `.autodoc.yml` creates `.autodoc/` directory
- Output contains README.md, per-directory AGENTS.md files, and diagrams/deps.mmd
- Running without a config file uses convention-based defaults
- CLI flags (`--exclude`, `--output-dir`) override config values
- Running on the gem's own source code produces self-documentation

**Edge Cases:**
- Empty directory: generates skeleton files with "No public symbols" placeholder
- Directory with no Ruby files: skips with warning, no crash
- Symlinked directories: follows links by default, flag to skip
- Very deep nesting (>10 levels): generates but truncates tree display
- Binary files in source tree: ignored by analyzer, not crash

### FR-2: Hierarchical Directory Indexing
**As an** AI agent,
**I want** an index file at every directory level showing file tree, module summaries, and cross-references,
**So that** I can navigate the codebase hierarchically without reading individual files.

**Acceptance Criteria:**
- Every directory in a module root gets an `_index.md` file in `.autodoc/`
- Each index includes: directory file tree, list of subdirectories with brief summaries, list of files with class/module names and public method counts, cross-reference links to sibling and parent directories
- Root-level index summarizes all module roots
- Indexes are regenerated on every `generate` run

**Edge Cases:**
- Single-file directory: index still generated, no crash
- Directory with only subdirectories (no files): lists subdirectory summaries only
- Directory with 100+ files: pagination not required, but listing is complete
- Hidden directories (starting with `.`): excluded by default, configurable

### FR-3: Semantic Search with Vector Embeddings
**As an** AI agent,
**I want** to query the documentation with natural language and get ranked results from vector search,
**So that** I can find relevant code, docs, and patterns without exact keyword matching.

**Acceptance Criteria:**
- `generate --embed` creates a local vector index (FAISS or LanceDB) in `.autodoc/vector/`
- Each chunk includes: function/method name, file path, docstring, code body (first 20 lines), scope chain, surrounding imports
- Embedding batches of 100 chunks for efficiency (45min for large codebases)
- Hybrid search: 60% semantic + 40% BM25 keyword, merged via Reciprocal Rank Fusion
- Query interface: `autodoc search "find user authentication logic"` returns ranked list with file paths, relevance scores, and snippet previews
- Index update is incremental: only re-embed changed files

**Edge Cases:**
- First run with no index: full embedding pass, progress feedback
- Query with no relevant results: returns empty result set, not error
- Very large codebase (10,000+ files): batch embedding, memory-bounded
- Non-text files in source: ignored for embedding
- Query contains special characters: sanitize before passing to vector search

### FR-4: Summary Generation per Module
**As a** developer,
**I want** auto-generated summaries for each module and directory describing its purpose and contents,
**So that** I can quickly understand what each part of the codebase does without reading source.

**Acceptance Criteria:**
- Each module root directory gets a `SUMMARY.md` in `.autodoc/`
- Summary includes: directory purpose (inferred from file names and structure), list of key classes/modules with one-line descriptions, list of public methods grouped by class, dependency overview (what this module imports and what imports it), doc coverage percentage
- Summaries are generated from analyzed code structure, not from AI/LLM
- Summaries include a "Last generated" timestamp

**Edge Cases:**
- Module with no doc comments: placeholder text "Undocumented — review and fill in purpose"
- Module with single class: still generates full summary structure
- Module with no public symbols: empty API surface section
- Renamed/moved directory: stale summary flagged on next `generate`

### FR-5: Rich Mermaid Diagrams
**As a** developer,
**I want** auto-generated Mermaid diagrams showing class hierarchies, ER relationships, architecture layers, and dependency graphs,
**So that** I can visualize the codebase structure at a glance.

**Acceptance Criteria:**
- `generate` produces at minimum: dependency DAG (`.autodoc/diagrams/deps.mmd`)
- When schema/migration files are present, produce ER diagram (`.autodoc/diagrams/er.mmd`)
- When class inheritance is detected, produce class hierarchy diagram (`.autodoc/diagrams/class.mmd`)
- Each diagram file includes Mermaid syntax that renders in GitHub/GitLab
- Diagrams include node labels with file counts or class names
- Styling: consistent color scheme per diagram type

**Edge Cases:**
- No inheritance relationships: skip class diagram, not empty file
- Circular dependencies: diagram renders with cyclic edge markers
- Very large graph (>50 nodes): split into subgraphs by namespace
- No schema/migration files: skip ER diagram silently

### FR-6: Grep/Keyword Search Integration
**As an** AI agent,
**I want** to search the documentation with exact keyword matching alongside semantic search,
**So that** I can find exact symbol names and file paths that semantic search might miss.

**Acceptance Criteria:**
- `autodoc search --keyword "User"` returns all files containing the exact keyword in documented symbols or file paths
- Results include: file path, line number, matching context line, symbol type
- Keyword search indexes: all class/module names, all method names, all file paths, all documented symbol names
- Results are ranked: exact matches first, partial matches second
- Combined with semantic search via Reciprocal Rank Fusion when both modes are active

**Edge Cases:**
- Case-insensitive search: matches "user", "User", "USER"
- Partial match: "auth" matches "authenticate", "authorize", "authentication"
- Special characters in query: escaped properly, no injection
- Empty result set: clear message, not error

### FR-7: Agent Query Interface
**As an** AI agent,
**I want** a structured query API exposed as a local CLI or HTTP endpoint that returns machine-parseable results,
**So that** I can programmatically ask questions about the codebase without shelling out to grep.

**Acceptance Criteria:**
- `autodoc query --find-symbol "User" --format json` returns JSON with file path, line number, type (class/module/method), doc string, and scope chain
- `autodoc query --find-dependents "app/models/user" --format json` returns all files that import or reference the given module
- `autodoc query --find-dependencies "app/models/user" --format json` returns all files that the given module imports
- `autodoc query --coverage --format json` returns per-directory and aggregate doc coverage stats
- `autodoc query --summary --dir "app/models" --format json` returns structured module summary
- All queries available as HTTP endpoints via the `serve` command: `/api/query?...`

**Edge Cases:**
- Invalid query type: returns error with valid options, not crash
- Symbol not found: returns `{"found": false, "suggestions": [...]}`
- Directory not documented: returns `{"status": "not_indexed"}`
- Concurrent queries: stateless, safe to run in parallel

### FR-8: Vector/Summary/Index Mapping
**As an** AI agent,
**I want** a single mapping file that correlates vector chunks to source files, summaries, and index locations,
**So that** I can jump from a search result to the right documentation file programmatically.

**Acceptance Criteria:**
- `.autodoc/map.json` is generated on every `generate --embed` run
- Map contains: for each vector chunk: chunk_id, source_file, doc_file (AGENTS.md/README.md path), summary_file path, line range, symbol name, scope path
- Map also contains: directory index file paths, diagram file paths, module root list
- Map uses a versioned schema (`schema_version: "1.0"`) for forward compatibility
- Map is valid JSON, parseable by any language

**Edge Cases:**
- No embeddings generated: map still created without vector sections
- File moved between runs: map rebuilds from scratch on full generate
- Corrupt map file: regenerate on next `generate` run, warn about previous corruption

### FR-9: Documentation Coverage Audit
**As a** CI pipeline,
**I want** a machine-parseable audit report with pass/fail thresholds,
**So that** I can gate pull request merges on documentation quality.

**Acceptance Criteria:**
- `autodoc audit --threshold 80` produces text table + `report.json`
- Exit code 0 when coverage >= threshold, exit code 1 when below
- `report.json` includes: total_symbols, documented, undocumented, coverage_percent, passed_threshold, failures list (file, symbol, type, line)
- Failures include: undocumented public symbols, modules exceeding max size
- Audit respects config file threshold unless overridden by CLI flag

**Edge Cases:**
- Threshold set to 0: always passes (for initial adoption)
- Threshold set to 100: only passes when every public symbol is documented
- Empty project (no symbols): passes by default (100% of 0 symbols)
- Project with only private methods: no public symbols to audit, passes

### FR-10: Orphan File Detection
**As a** developer,
**I want** to find files that have no import edges and are not referenced in generated docs,
**So that** I can identify dead or disconnected code.

**Acceptance Criteria:**
- `autodoc orphans <path>` lists files with zero inbound and zero outbound import edges
- Results include: file path, file size, last modified time, number of defined symbols
- Orphan analysis cross-references import graph with documented file references
- Results exclude test files, spec files, and fixture directories by default
- Exit code 0 when no orphans found, exit code 2 when orphans exist

**Edge Cases:**
- Entry point files (main executable, config.ru, Rakefile): flagged with "(entry point)" annotation
- Files that import nothing but are imported by others: NOT orphans
- Empty directory: no orphans, informative message
- All files are orphans (new project): lists all, exit code 2

## Non-Functional Requirements

| ID | Requirement | Target | Measurement |
|----|-------------|--------|-------------|
| NFR-1 | Generation speed | < 30s for 500-file project | `time autodoc generate` |
| NFR-2 | Embedding speed | < 45min for 10,000 files | Batch embedding with 100 chunks/batch |
| NFR-3 | Query response time | < 500ms for search results | `time autodoc search "query"` |
| NFR-4 | Spec coverage | >= 90% line coverage | `simplecov` report |
| NFR-5 | Test count | >= 70 passing RSpec examples | `bundle exec rspec --format progress` |
| NFR-6 | CLI exit codes | Correct exit codes per subcommand | Integration test assertions |
| NFR-7 | No external AI dependency | Summaries from code analysis, not LLM | Review generated summaries |
| NFR-8 | Output portability | All outputs valid Markdown + JSON | Schema validation on output |

## Out of Scope

- AI/LLM-generated documentation text (summaries are structural, not generative)
- Multi-language support beyond Ruby (deferred to Phase 3)
- Live documentation watching / file-system watcher (deferred)
- Cloud-hosted documentation server (deferred)
- Documentation diff preview in browser (deferred)
- Third-party integration with Notion, Confluence, etc. (deferred)
- Automated docstring generation (writes placeholders, not content)
- IDE plugins or editor extensions (deferred)
